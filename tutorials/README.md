# Tutorials

A step-by-step set of tutorials for building voice and multimodal agents with Pipecat. The examples are ordered by number, with each new file adding more capability than the last. Files ending in `a` are **local-execution variants** that bypass Daily/WebRTC and use your PC's microphone, speaker, or a Tkinter window directly.

## How to run

Run the examples from the project root via `uv`:

```bash
# Launch with a selectable transport (Daily / WebRTC / Twilio)
uv run python tutorials/01-say-one-thing.py

# Specify a transport (e.g. WebRTC so you can connect from a browser)
uv run python tutorials/06-voice-agent.py -t webrtc
```

The `a`-suffixed files (`01a`, `03a`, `06a`) use local devices directly, so no transport flag is needed:

```bash
uv run python tutorials/01a-local-audio.py
```

## Required environment variables

Add the API keys used by each tutorial to a `.env` file. The required keys differ per file.

| Variable | Purpose | Used by |
|----------|---------|---------|
| `CARTESIA_API_KEY` | TTS (speech synthesis) | 01, 01a, 02, 04, 05, 06, 06a, 07 |
| `ANTHROPIC_API_KEY` | LLM (Claude) | 02 |
| `OPENAI_API_KEY` | LLM (GPT) | 04, 05, 06, 06a, 07 |
| `GOOGLE_API_KEY` | Image generation (Gemini/Imagen) | 03 |
| `FAL_KEY` | Image generation (fal.ai) | 03a, 04 |
| `DEEPGRAM_API_KEY` | STT (speech recognition) | 05, 06, 06a, 07 |

## 01 — `01-say-one-thing.py`

**Goal:** The smallest possible Pipecat pipeline. When a client connects, the bot speaks a single line and exits.

**Pipeline:**
```
TTS → transport.output()
```

**Key points:**
- Uses `CartesiaTTSService` to synthesize "Hello there!".
- The `on_client_connected` handler queues a `TTSSpeakFrame` followed by an `EndFrame`, so the pipeline shuts down after the utterance finishes.
- `transport_params` registers three transports (daily / twilio / webrtc), selectable at launch time.

---

## 01a — `01a-local-audio.py`

**Goal:** Same behavior as `01`, but the audio is played through the **local speaker** instead of a network transport.

**Differences:**
- Uses `LocalAudioTransport` and is launched directly with `asyncio.run(main())`.
- After a 1-second delay, the `say_something()` coroutine queues a `TTSSpeakFrame`.
- Handy for verifying your audio device works without needing Daily/WebRTC connectivity.

---

## 02 — `02-llm-say-one-thing.py`

**Goal:** Extend `01` with an LLM so the spoken line is **generated by the model** rather than a fixed string.

**Pipeline:**
```
LLM → TTS → transport.output()
```

**Key points:**
- Uses `AnthropicLLMService` (Claude). Commented-out blocks show how to swap in `OpenAILLMService` or `GoogleLLMService`.
- An `LLMContext` is seeded with `{"role": "developer", "content": "Say hello to the world."}` and pushed as an `LLMContextFrame`, so the LLM generates a response → TTS speaks it → transport plays it.
- The system prompt explicitly tells the model "your responses will be spoken aloud, so avoid emojis and bullet points" — a practical touch for voice UX.

---

## 03 — `03-still-frame.py`

**Goal:** Pipe **generated images** to the transport's video output. Three stills are produced in sequence.

**Pipeline:**
```
ImageGen (Google) → transport.output()
```

**Key points:**
- Three prompts (`TextFrame`) are queued to `GoogleImageGenService`: a Picasso-style cat, dog, and fish.
- Transport is configured with `video_out_enabled=True` at `1024×1024`.
- `PipelineParams(enable_metrics=True, enable_usage_metrics=True)` exposes inference latency and usage metrics.
- The `on_client_disconnected` handler calls `task.cancel()`, demonstrating the correct way to release resources when the client leaves.

---

## 03a — `03a-local-still-frame.py`

**Goal:** A local version of `03` that renders images into a **Tkinter window**.

**Differences:**
- Uses `TkLocalTransport` with a `tk.Tk()` root as the display target.
- Image generation is handled by `FalImageGenService` (fal.ai); an `aiohttp.ClientSession` must be passed explicitly.
- A `run_tk()` coroutine calls `tk_root.update()` every 100 ms so the GUI stays responsive, running alongside the pipeline via `asyncio.gather`.

## 04 — `04-sync-speech-and-image.py`

**Goal:** A calendar-narration bot: the LLM describes a scene for each month of the year, and the **spoken audio is synchronized with a matching generated image**. This is where the tutorials step up in complexity.

**Pipeline:**
```
LLM
  → SentenceAggregator
  → SyncParallelPipeline(
        [ ImageGen → MarkImageForPlaybackSync ],
        [ MonthPrepender → TTS ],
        frame_order=FrameOrder.PIPELINE)
  → transport.output()
```

**Key points:**
- **Custom frame:** `MonthFrame`, a `DataFrame` that carries the current month name.
- **Custom processors:**
  - `MarkImageForPlaybackSync` — flags every outgoing image frame with `sync_with_audio = True` so playback aligns image and speech.
  - `MonthPrepender` — prepends "January: …" to the first `TextFrame` after each `LLMFullResponseStartFrame`.
- **`SyncParallelPipeline`:** runs image generation and TTS concurrently while preserving ordering (image for month N → audio for month N → image for month N+1 …). `FrameOrder.PIPELINE` emits sub-pipeline outputs in declaration order: image first, then audio.
- The final stage of each branch must be synchronous, so HTTP-based services (`FalImageGenService`, `CartesiaHttpTTSService`) are used rather than streaming ones.
- `TextAggregationMode.TOKEN` prevents the TTS service from waiting for sentence completion (which would otherwise break the sync mechanism).

## 05 — `05-speaking-state.py`

**Goal:** A two-way voice conversation with a minimal avatar: the displayed image **switches between "speaking" and "waiting" states** depending on whether the bot is talking.

**Pipeline:**
```
transport.input() → STT → user_aggregator
  → LLM → TTS
  → ImageSyncAggregator
  → transport.output()
  → assistant_aggregator
```

**Key points:**
- **STT added:** `DeepgramSTTService` transcribes the user's speech.
- **VAD:** `SileroVADAnalyzer` is passed via `LLMUserAggregatorParams` to detect utterance boundaries.
- **`ImageSyncAggregator` (custom):** on `BotStartedSpeakingFrame`, pushes `speaking.png` as an `OutputImageRawFrame`; on `BotStoppedSpeakingFrame`, pushes `waiting.png`. Requires `assets/speaking.png` and `assets/waiting.png` in the repo.
- **Aggregator pair:** `LLMContextAggregatorPair` accumulates both user and assistant turns into the same `LLMContext`, preserving conversational memory.

## 06 — `06-voice-agent.py`

**Goal:** A **full voice agent** without visuals (STT + LLM + TTS). Think of this as the practical baseline stripped of the image work from `05`.

**Pipeline:**
```
transport.input() → STT → user_aggregator
  → LLM → TTS
  → transport.output()
  → assistant_aggregator
```

**Key points:**
- On client connect, the conversation is kicked off with `context.add_message({"role": "developer", "content": "Please introduce yourself to the user."})` followed by an `LLMRunFrame()`.
- Supports daily / twilio / webrtc transports — Twilio in particular makes this extendable to a phone-based answering bot.
- This is the **standard template** for a Pipecat voice agent.

---

## 06a — `06a-voice-agent-local.py`

**Goal:** Local-device version of `06`. Talk to the bot directly through your PC's microphone and speaker.

**Differences:**
- Uses `LocalAudioTransport(audio_in_enabled=True, audio_out_enabled=True)`.
- No transport selector — launched straight from `asyncio.run(main())`.
- Doesn't wait for a connection event; the `LLMRunFrame()` is queued immediately at startup to begin the conversation.

---

## 07 — `07-function-calling.py`

**Goal:** Add **LLM function calling (tool use)** to the voice agent.

**Pipeline:** Same as `06` (STT + LLM + TTS), plus tool definitions and function registration.

**Key points:**
- **Tool definitions:** `FunctionSchema` declares `get_current_weather` (location, format) and `get_restaurant_recommendation` (location); both are bundled into a `ToolsSchema` and passed to `LLMContext(tools=tools)`.
- **Function registration:** `llm.register_function("get_current_weather", fetch_weather_from_api)` binds an async callback that the LLM invokes. The callback returns its result via `params.result_callback(...)`.
- **UX touch:** an `on_function_calls_started` handler pushes `tts.queue_frame(TTSSpeakFrame("Let me check on that."))` so the bot doesn't go silent while the function runs.
- The sample callbacks return hard-coded values ("nice, 75°", "The Golden Dragon"). In a real deployment you'd swap them for weather APIs, RAG/DB lookups, or other services.

## Suggested learning path

1. **01 → 01a:** Grasp the Pipeline / Frame / Transport model.
2. **02:** Learn how to plug in an LLM and the basics of `LLMContext`.
3. **03 → 03a:** Add image generation and experience a video-output transport.
4. **04:** Apply custom `FrameProcessor`s and `SyncParallelPipeline` for synchronized output.
5. **05:** Two-way conversation plus visual state changes — the first real step toward embodied behavior.
6. **06 → 06a:** The bread-and-butter voice-agent template.
7. **07:** Add external actions (function calling) to turn the agent into something that can *do* things.
